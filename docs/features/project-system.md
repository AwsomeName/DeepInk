# DeepInk 工作空间系统 — 产品信息架构

> 状态：📋 规划中
> 优先级：近期产品架构调整
> 目标：将现有“文件区”升级为“工作空间区”，让工作空间成为 DeepInk 的核心工作组织单位。

相关体验页面边界见 `docs/features/product-experience-pages.md`。本文只聚焦工作空间系统本身。

## 一句话原则

DeepInk 以 Home 为入口，以工作空间组织工作，以标签页承载当前操作，以文件、草稿、会话和记忆沉淀结果。

## 核心术语

| 术语         | 面向用户的含义                 | 系统含义                                                               |
| ------------ | ------------------------------ | ---------------------------------------------------------------------- |
| Home         | 总入口、默认界面、继续工作页   | 聚合最近工作空间、未归档内容、待确认任务和快速开始动作                 |
| 工作空间     | 一件工作事项的容器             | 本地文件夹、远程目录、未来无文件夹工作空间                             |
| 工作空间来源 | 工作空间的文件与执行环境来源   | 本地、远程设备/服务器、系统未归档                                      |
| 连接通道     | 远程工作空间如何连上           | 直连、CCLink、未来 VPN/Gateway 等 transport                            |
| 执行后端     | 当前工作空间由谁执行任务       | DeepInk Agent、Codex、Claude Code、自定义后端                          |
| 工作空间区   | 左侧管理工作空间的区域         | 由现有文件区升级而来，包含工作空间列表、激活工作空间内容、文件树和会话 |
| 标签页       | 当前工作空间里打开的工作项     | 浏览器、Markdown、Android、Terminal、预览、设置、会话等 Tab            |
| 文件         | 正式保存的成果或资料           | 本地文件、同步文件、导出文件                                           |
| 草稿         | 尚未保存到具体文件的内容       | Markdown 临时文档、Agent 生成但未归档的内容                            |
| 会话         | 用户与 AI 的协作过程           | Agent 对话、工具调用记录、任务状态与摘要                               |
| 未归档       | 没有明确归属工作空间的临时区域 | 承接未归档内容、临时草稿、全局会话和待整理内容                         |
| 系统工作空间 | DeepInk 内置的隐藏存储归属     | 承接未归档内容、用户长期记忆和 App 自有资料                            |

不再新增抽象产品概念；用户只需要理解“工作空间、标签页、文件、草稿、会话、未归档”。

## 命名约束

DeepInk 的产品术语和代码术语保持分层：

```text
用户看到：工作空间
代码实现：workspace
```

也就是说，产品文案、按钮、侧栏和帮助说明中优先称为“工作空间”；“项目”只作为旧称或口语同义词。工程代码中继续沿用现有 `workspacePath`、`WorkspaceStateService`、`workspace-state` 等命名。

短期不新增另一套核心服务，也不把现有 `workspace` 全量重命名。这样可以避免在产品概念还在演进时制造两套并存模型。

约束：

- UI 文案优先使用“工作空间”，必要时可在过渡期使用“项目”作为同义词。
- 代码层继续使用 `workspace`，表示“工作空间”的当前实现。
- 文档中第一次涉及工程实现时说明“工作空间在代码中由 workspace 表示”。
- 未来如果支持无文件夹工作空间，也优先扩展 workspace 模型，而不是另起一套并行模型。

```text
│   ├── 未归档草稿
│   ├── 待确认任务
│   └── 快速开始
│
├── 工作空间
│   ├── 文件
│   ├── 草稿
│   ├── 会话

```

## Home

Home 是 DeepInk 的默认入口，不是一个工作空间，也不是空状态页面。

Home 回答三个问题：

- 我现在要从哪里继续？
- 还有什么没有处理？
- 我可以快速开始什么？

Home 展示：

| 区域     | 内容                                                      |
| -------- | --------------------------------------------------------- |
| 继续工作 | 最近工作空间、最近标签页、最近会话、最近草稿              |
| 未处理   | 未保存草稿、未归档会话、Agent 等待确认、失败任务          |
| 快速开始 | 新建 Markdown、打开网页、新建 AI 会话、打开工作空间文件夹 |
| 个人资料 | 记忆更新建议、常用偏好入口、临时内容整理入口              |

Home 的数据主要来自系统工作空间和最近用户工作空间。

## 工作空间区

现有文件区升级为工作空间区。工作空间区的主体不是文件树，而是工作空间列表；文件树只是激活工作空间的展开内容。

工作空间区结构参考 Codex，但保留 DeepInk 的文件浏览能力：

```text
工作空间区
├── 工作空间
│   ├── [本地] GAME
│   ├── [本地] DeepInk                  ← 激活时展开
│   │   ├── 文件                        ← 本地文件树
│   │   └── 会话                        ← 当前工作空间会话，可折叠
│   ├── [远程 · 直连] Mac mini /Users/app/project-a
│   └── [远程 · CCLink] supermicro /data/research ← 激活时展开
│       ├── 文件                        ← 远程文件树
│       └── 会话                        ← 当前工作空间会话，运行在远端
│
└── 未归档                               ← 固定底部
    ├── 未归档会话
    └── 临时草稿
```

用户选择某个工作空间后：

- 左侧在该工作空间行下展开文件树和会话。
- 中间恢复该工作空间自己的标签页集合。
- 右侧 Agent 围绕当前工作空间和当前标签页协作。

约束：

- 本地和远程不作为一级分组，只作为工作空间 badge，降低层级复杂度。
- 直连、CCLink、VPN 等只表示连接通道，不新增一级目录。
- 同一时间只展开当前激活工作空间，避免多个文件树和会话列表同时撑爆侧栏。
- Tab 列表不进入侧栏，仍由主工作区顶部 Tab Bar 管理，并随工作空间激活状态切换。
- 未归档不伪装成普通工作空间，固定在侧栏底部，承接没有工作空间归属的会话和草稿。
- 不设计单独的“远程 Agent 面板”；远程会话、远程文件和远程 Terminal 都进入对应远程工作空间。

添加入口：

```text
添加工作空间
├─ 打开本地文件夹
├─ 添加远程工作空间
│  ├─ 直连服务器
│  └─ 通过 CCLink
└─ 新建临时草稿
```

添加成功后，工作空间出现在同一层工作空间列表中；用户不需要停留在连接管理页。

## 工作空间来源与 WorkspaceRef

工作空间必须从“本地文件夹”扩展为“工作归属”，以支持 VSCode Remote 类场景、直连远程服务器和 CCLink 远程工作区。

产品层：

```text
工作空间 = 一个可承载文件、会话、Tab 状态的工作归属
```

Remote 是 DeepInk 的远程工作能力，CCLink 只是 Remote 的连接通道之一。直连 Remote 与 CCLink Remote 在工作空间列表中平级，区别只体现在 badge、副标题和连接设置中。

工程层建议引入通用引用，而不是继续把所有工作空间都塞进 `workspacePath: string | null`：

```ts
type WorkspaceRef =
  | { kind: 'local'; path: string }
  | {
      kind: 'remote'
      transport: 'direct' | 'cclink'
      endpointId: string
      workspaceId: string
      path: string
      label?: string
    }
  | { kind: 'global' }
```

第一阶段仍保留 `workspacePath` 兼容本地工作空间；后续新增 `activeWorkspaceRef` / `workspaceRef` 作为统一入口。

状态持久化层使用 `workspaceKey` 隔离不同工作空间：

- 本地工作空间：`workspaceKey = /Users/.../DeepInk`，兼容旧 `workspacePath`。
- 直连远程工作空间：`workspaceKey = direct://endpointId/pathHash`。
- CCLink 远程工作空间：`workspaceKey = cclink://serverId/workspaceId`。
- 未归档：`workspaceKey = null`，对应 global workspace。

`WorkspaceStateSnapshot.workspacePath` 短期保留为兼容字段；新逻辑应优先使用 `workspaceKey`。

| 工作空间来源  | 示例                                          | 文件 Provider          | 会话运行位置                  |
| ------------- | --------------------------------------------- | ---------------------- | ----------------------------- |
| 本地          | `[本地] DeepInk`                              | Electron 主进程 `fs`   | 本机                          |
| 远程 · 直连   | `[远程 · 直连] Mac mini /Users/app/project-a` | Direct remote provider | 远端机器，直连 transport      |
| 远程 · CCLink | `[远程 · CCLink] supermicro /data/research`   | CCLink file service    | 远端机器，CCLink transport    |
| 未归档        | `未归档`                                      | 无文件树               | 全局 / 临时，尚未明确工作空间 |

文件树 UI 只依赖统一 provider：

```ts
interface WorkspaceFileProvider {
  listDir(workspaceRef, path): Promise<FileEntry[]>
  readFile(workspaceRef, path): Promise<FileContent>
  writeFile(workspaceRef, path, content): Promise<void>
}
```

本地 provider 复用当前 `fs` IPC；直连远程 provider 通过用户配置的远端地址/凭证访问；CCLink provider 通过 CCLink 和 `chatcc-agent` 访问远程目录。

## Remote、CCLink、Codex 与 Terminal

这几个概念必须分开，否则产品会越做越像调试工具：

| 概念        | 正确定位                                      | 不能混成                          |
| ----------- | --------------------------------------------- | --------------------------------- |
| Remote      | 工作空间可以在远端机器上                      | 不等同 CCLink                     |
| CCLink      | Remote 的一种 transport                       | 不等同远程工作空间本身            |
| 直连 Remote | 用户自有 IP/域名/内网/VPN/SSH/远端 Agent 地址 | 不需要 CCLink 账号配对            |
| Codex       | 执行后端/Agent backend                        | 不等同工作空间类型                |
| Terminal    | 当前工作空间里的 Tab 类型                     | 不放 Activity Bar，不做单独产品区 |
| 会话        | 用户与 AI/Agent 的协作过程                    | 不分裂成本地会话/远程 Agent 面板  |

因此：

- 用户添加远程时，应该先选择“添加远程工作空间”，再选择连接方式：直连或 CCLink。
- 工作空间列表只展示工作空间，不展示“服务器父目录”。
- 远程工作空间的执行后端可选择 Codex、DeepInk Agent、Claude Code 或自定义后端。
- 会话统一叫“会话”；本地和远程只通过运行位置、连接通道、执行后端和状态区分。
- 本地 Terminal 运行本机 shell，远程 Terminal 运行远端 shell，并继承当前工作空间的权限、生命周期和审计规则。
- Terminal 的加入不意味着 DeepInk 变成 IDE；它是 AI 执行和远程运维的可视化现场。

会话元信息：

- 运行位置：本机 / Mac mini / supermicro。
- 连接通道：local / direct / cclink。
- 执行后端：DeepInk Agent / Codex / Claude Code / 自定义后端。
- 状态：在线 / 离线 / 执行中 / 等待确认。

## 工作空间与标签页

工作空间绑定自己的标签页状态。

```text
激活工作空间 A
→ 打开的浏览器、Markdown、Android、预览、会话 Tab 都属于工作空间 A

切换到工作空间 B
→ 保存工作空间 A 的标签页快照
→ 隐藏工作空间 A 的标签页
→ 恢复工作空间 B 的标签页
```

用户感知是“切换工作空间”，不是“关闭标签页”。

标签页分为两类：

| 类型           | 归属                                                  |
| -------------- | ----------------------------------------------------- |
| 工作空间标签页 | Markdown、浏览器、Android、Terminal、预览、会话、草稿 |
| 全局标签页     | Home、设置、账号、订阅、全局消息                      |

全局标签页不随工作空间切换消失。

## 系统工作空间

系统工作空间是 DeepInk 内置的隐藏工作空间，不出现在普通工作空间列表中。它解决“没有工作空间时内容放哪里”的问题。

系统工作空间承接：

- 没有激活工作空间时新建的 Markdown 草稿。
- 没有归属工作空间的 AI 会话。
- Agent 生成但用户尚未决定保存位置的内容。
- 临时下载、截图、附件、浏览器任务产物。
- 用户长期记忆、偏好、常用信息和全局上下文摘要。

系统工作空间分为两类区域：

| 区域                  | 生命周期 | 说明                                           |
| --------------------- | -------- | ---------------------------------------------- |
| Memory                | 长期     | 用户可查看、编辑、删除的正式记忆               |
| Inbox / Drafts / Temp | 临时     | 可移动到工作空间、保存为文件、归档为记忆或丢弃 |

记忆写入必须用户可控。临时内容不能自动变成长期记忆。

## Markdown 草稿

DeepInk 支持类似 macOS 便笺的 Markdown 所见即所得草稿体验。

流程：

```text
新建 Markdown 草稿
→ 默认归属当前工作空间
→ 若没有当前工作空间，则进入系统工作空间 Drafts
→ 自动快照，防止崩溃丢失
→ 可打开多个草稿标签页
→ 关闭时选择：保存为文件 / 保留草稿 / 丢弃
```

保存为文件后，草稿变成工作空间文件。保留草稿时继续出现在 Home 和工作空间区的草稿分组中。

## 与现有实现的对应关系

当前代码已经具备工作空间系统的基础：

| 当前实现             | 工作空间系统含义           |
| -------------------- | -------------------------- |
| `workspacePath`      | 当前工作空间路径           |
| `WorkspaceState`     | 工作空间状态快照           |
| `tabs`               | 工作空间内标签页           |
| `browserTabs`        | 工作空间内浏览器状态       |
| `editorDrafts`       | 工作空间内草稿和未保存文件 |
| `agentConversations` | 工作空间内 AI 会话         |
| `fileTree`           | 工作空间内文件树状态       |

第一阶段不需要推倒重来，而是把现有“工作区文件夹”正式产品化为“工作空间”。

系统工作空间第一版复用 `workspacePath: null` 对应的 global workspace：

| 工程概念                | 产品含义                    |
| ----------------------- | --------------------------- |
| `workspaceId: "global"` | 隐藏系统工作空间            |
| `workspacePath: null`   | 未归档 / 没有绑定本地文件夹 |
| `editorDrafts`          | 系统草稿 / 未归档草稿       |
| `agentConversations`    | 全局会话 / 未归档会话       |

系统工作空间不作为普通工作空间管理，不支持重命名、删除或同步；但工作空间区底部固定展示“未归档”，让用户可以进入全局会话和临时草稿。

## 分阶段实现

### Phase 1：术语和入口调整

- 将“文件区”命名和文案升级为“工作空间区”。
- 增加 Home 入口和默认 Home 页面。
- 工作空间第一版等同本地文件夹，复用当前 `workspacePath`。
- 工作空间区显示最近工作空间、打开工作空间文件夹、当前工作空间文件树。
- 保持代码中的 `workspace` 命名不变，先完成产品文案和信息架构收敛。
- 保持 Activity Panel 的内部 ID 为 `files`，产品 UI 显示为“工作空间”；稳定后再评估是否迁移内部 ID。

### Phase 1.5：工作空间区外壳

- 在工作空间区顶部展示当前工作空间名称、工作空间路径和当前工作空间标签页数量。
- 在工作空间区增加“最近工作空间”分组，第一版先展示当前工作空间作为占位。
- 在工作空间区增加“当前工作空间文件”分组，继续复用现有 `FileTree`。
- 不新增另一套数据层；真正的最近工作空间列表放到 Phase 2 随 workspace 切换一起实现。

### Phase 1.6：草稿与会话分组

- 在工作空间区增加“草稿”分组，第一版读取现有 `editorDrafts` / `editor-store.files` 中的虚拟文档和 dirty 文件。
- 在工作空间区增加“会话”分组，第一版读取现有 `agentConversations` / `agent-store.conversations` 中的最近会话。
- 会话分两种展示形态：右侧 Agent 面板承接即时助手会话，工作空间 `会话` 分组承接可恢复的工作会话；远程工作空间会话默认打开为 `conversation` Tab。
- 草稿第一版先展示状态，不承担完整恢复、移动或归档。
- 不改变草稿和会话的数据归属模型，完整的工作空间绑定与系统工作空间归档放到 Phase 2/3。

### Phase 2：工作空间状态隔离

- 在设置中维护 `recentWorkspacePaths`，产品侧显示为“最近工作空间”。
- 打开或切换工作空间时更新最近工作空间列表，保持 `lastWorkspacePath` 与最近工作空间顺序一致。
- 每个工作空间拥有独立的 `tabs`、`browserTabs`、`editorDrafts`、`agentConversations`、`fileTree`。
- 切换工作空间时保存当前工作空间快照并恢复目标工作空间快照。
- 目标工作空间没有快照时，清空当前标签页、浏览器状态、草稿和会话，避免继承上一个工作空间现场。
- 全局标签页不参与工作空间状态切换。
- 优先补齐最近 workspace 列表和 workspace 切换体验，不引入新的并行数据层。

### Phase 2.5：全局标签页边界

- 代码中集中维护工作空间标签页判断，第一版 `settings` 视为全局标签页。
- WorkspaceState 的 `tabs` section 只保存工作空间标签页。
- 恢复工作空间标签页时保留当前全局标签页，避免设置页等全局页面被工作空间切换关闭。
- 后续新增 Home、账号、订阅等 Tab 时，默认归入全局标签页，不进入工作空间快照。

### Phase 2.6：草稿切换保护

- 切换工作空间或打开新的工作空间文件夹前，检查当前工作空间是否存在 dirty 编辑器文件或未命名虚拟草稿。
- 如果存在未保存草稿，先提示用户：草稿会保留在当前工作空间，稍后切回可继续。
- 提示前主动镜像一次 `editorDrafts` 到当前 workspace，降低异步持久化窗口。
- 第一版只做“保留并切换 / 取消切换”，不做保存为文件、丢弃、移动到工作空间等完整草稿治理。

### Phase 3：系统工作空间

- 复用 global workspace 作为隐藏系统工作空间。
- 没有当前工作空间时，新建草稿、会话和 Agent 产物进入系统工作空间。
- 工作空间区底部固定展示“未归档”，参考 Codex 的未归档会话区域。
- 点击未归档会话或草稿时，当前归属自然切到 global workspace；UI 不强调“退出工作空间”。
- 未归档区域展示全局会话和临时草稿，不展示文件树。
- Home 展示系统工作空间中的未归档内容。
- 支持“移动到工作空间”“保存为文件”“丢弃”。

### Phase 3.0：工作空间区信息治理

工作空间区不能把所有对象平铺在同一个层级，否则用户无法判断“我在哪里”和“这些东西归谁”。第一版按三层治理：

1. **工作空间列表层**：工作空间区首先是工作空间列表。本地工作空间和远程工作空间平铺展示，用 badge 区分来源，激活项高亮。
2. **激活工作空间内容层**：只有当前工作空间在工作空间行下内联展开文件和会话。
3. **未归档层**：固定在侧栏底部，承接未归属会话、临时草稿和待整理内容。

治理原则：

- 工作空间列表永远只放工作空间，不塞文件、不塞草稿。
- 本地和远程只是来源标识，不作为一级分组。
- 同一时间只展开一个激活工作空间，展开内容包括文件树和会话列表。
- 未归档固定底部，不参与文件树，不作为可删除/重命名的普通工作空间。
- 高频入口可以快捷化，但主入口必须语义完整；例如 Tab 栏 `+` 是新建标签页菜单，浏览器图标只是快捷入口。

### Phase 3.1：草稿恢复

- 工作空间区和未归档视图中的草稿列表可点击。
- 已打开的虚拟草稿会激活对应 Tab。
- 已关闭的虚拟草稿会用保存的 Markdown 内容重新打开为未命名编辑器 Tab。
- 已关联文件路径的 dirty 草稿会打开对应文件 Tab。
- 第一版只做打开/恢复，不做移动、丢弃或批量整理。

### Phase 3.2：草稿生命周期

草稿不是独立产品概念，而是工作空间里的未归档文档状态。它解决的是“先写，稍后决定保存位置”的便笺式工作流。

**新建入口**：

- 无工作空间时，工作空间区展示“未归档”系统工作空间，点击“新建 Markdown 草稿”创建系统草稿。
- 有工作空间时，工作区 Tab 栏的“新建文档”按钮创建当前工作空间草稿。
- 后续可在工作空间区“草稿”分组标题右侧增加 `+`，作为更明确的草稿入口。
- Agent 写文档且没有明确文件路径时，也进入当前工作空间或未归档系统工作空间的草稿。

**归属规则**：

- 当前打开本地工作空间时，新草稿归属当前工作空间。
- 没有打开工作空间时，新草稿归属隐藏系统工作空间，即“未归档”。
- 用户点击未归档会话或草稿时，当前归属切到未归档，让之后的新草稿和会话进入系统工作空间。
- 切换工作空间不会把草稿带走；草稿留在创建它的工作空间里。
- 用户主动“保存到文件”后，草稿变成普通工作空间文件，不再作为未命名草稿管理。

**关闭规则**：

- 空白未命名草稿：关闭 Tab 时静默丢弃。
- 有内容的未命名草稿：关闭 Tab 时提供“保存到文件 / 保留草稿 / 丢弃”。
- 选择“保存到文件”：弹出保存位置选择，写入 Markdown 文件，然后关闭 Tab 并移除草稿。
- 选择“保留草稿”：只关闭 Tab，草稿留在工作空间区“草稿”列表中，可稍后恢复。
- 选择“丢弃”：关闭 Tab 并删除草稿状态。
- 已关联文件路径且有未保存修改：关闭 Tab 时提供“保存 / 不保存 / 取消”。

### Phase 3.3：标签页新建入口

- Tab 栏的 `+` 是“新建标签页”菜单，不绑定单一 Markdown 文档。
- 第一版菜单中可直接创建：Markdown 草稿、浏览器页、Android 页。
- 浏览器图标保留为高频快捷入口，等价于直接新建浏览器页。
- Word、PPT、表格等 Office 类型在文档能力完成前只展示为“规划中”，不创建假 Tab。
- 后续接入 Office/预览能力后，再把对应类型从禁用项升级为可创建项。

### Phase 4：远程工作空间与连接通道

- 工作空间列表支持本地工作空间和远程工作空间平铺展示。
- 远程工作空间来源可以是直连服务器，也可以是 CCLink 设备/服务器上的 workspace。
- 远程工作空间展示格式：`[远程 · 通道] 设备名 / 路径`，路径作为副标题或 tooltip。
- 激活远程工作空间时，恢复该远程工作空间自己的 Tab 状态、文件树和会话。
- 文件树走统一 `WorkspaceFileProvider`；本地 provider 使用 `fs`，直连 provider 使用 direct remote service，CCLink provider 使用 CCLink file service。
- 会话列表走统一工作空间归属；本地和远程都展示为 `会话`，运行位置和连接通道只作为元信息。
- 第一阶段远程工作空间可成为当前工作归属，并使用 `workspaceKey` 隔离 Tab/会话/草稿状态。
- 远程工作空间激活后，侧栏先展示远程文件树或明确的不可用原因，不伪造成空目录。
- CCLink 账号、导入旧账号、配对、服务器同步和诊断进入 Settings，不占用 Activity Bar。
- 不新增“远程 Agent 面板”；如果需要打开远程协作过程，打开的是对应工作空间里的 `会话` Tab。

**迁移顺序**：

1. 先清日常侧栏入口：把现有 CCLink、Android 管理、同步配置和诊断能力迁到 Settings 或对应 Tab，工作空间面板只展示工作空间内容。
2. 让 CCLink 已配对服务器生成远程工作空间条目，格式保持和本地工作空间平级。
3. 抽象 `WorkspaceFileProvider`，把本地文件树和 CCLink 远程文件树收敛到同一渲染入口。
4. 抽象工作空间会话列表，让本地会话、CCLink 远程会话、未来直连 Remote 会话使用同一列表。
5. 补直连 Remote 的数据模型和添加入口，但真实协议可以后置。

**第一步重点**：

- Activity Bar 不再把 CCLink、远程 Agent、Android 管理、同步配置当成一级入口。
- Settings 接管账号、连接通道、设备、同步和诊断。
- 工作空间面板只关心“有哪些工作空间，以及当前工作空间里有什么”。
- Workbench Tab 接管 Android、Browser、Markdown、Remote File、Session 和未来 Terminal 的运行态。
- Status Bar 只显示关键状态和轻量入口，不变成第二个设置页。

**迁移验收**：

- Activity Bar 不出现 CCLink/远程 Agent/Android 管理入口。
- 远程服务器不作为一级目录，只作为远程工作空间来源。
- 一个远程工作空间至少能展示在线状态、文件树不可用原因、会话归属和 Tab 快照。
- 用户能从 `添加工作空间` 进入本地、直连 Remote、CCLink 三种来源，而不是去找不同侧栏。
- 设置页能完成账号、配对、同步服务器、日志和缓存清理；日常侧栏不再承担运维台职责。

### Phase 4.5：Terminal Tab

- Tab 栏新增 Terminal 类型，跟随当前工作空间归属。
- 本地工作空间打开本机 shell；远程工作空间打开远端 shell。
- Terminal 会话必须记录工作空间、连接通道、执行后端和权限模式。
- 远程 Terminal 默认需要权限确认，不能继承本机宽权限。
- 关闭 Terminal Tab 时要定义进程生命周期：保持后台运行、发送中断、结束会话或转为任务记录。

### Phase 5：记忆系统接入

- 系统工作空间增加 Memory 分区。
- 用户可查看、编辑、删除长期记忆。
- Agent 写入长期记忆前需遵循用户授权和可审计规则。
- 工作空间会话可被总结为工作空间摘要，但不默认污染全局记忆。

## 设计约束

- 不要求用户先建工作空间才能开始工作。
- 不把工程代码中的 `workspace` 过早重命名。
- 不把工作空间区做成单纯文件树。
- 不让切换工作空间表现为关闭标签页。
- 不让临时草稿自动变成长期记忆。
- 不让系统工作空间作为普通工作空间被误删、重命名或同步污染。
- 不把 CCLink 当作 Remote 的唯一实现。
- 不把 Codex 当作工作空间类型。
- 不把 Terminal 做成无归属、无权限边界的全局入口。
- 不把远程会话特殊化成单独入口或单独面板。

## Grilling

1. 工作空间第一版等同文件夹很稳，但会限制无文件夹场景；系统工作空间必须兜底。
2. 工作空间区同时承载工作空间列表和文件树，层级必须清楚：先选工作空间，再看工作空间内容。
3. Home 和工作空间区不能职责重叠：Home 负责继续和待处理，工作空间区负责管理工作空间内对象。
4. 切换工作空间时必须保存和恢复标签页状态，否则用户会觉得工作现场丢失。
5. 记忆和临时内容必须分区，否则系统工作空间会变成不可审计的黑盒。
6. `workspace`、工作空间和“项目”（旧称/同义词）必须保持一一解释关系；如果代码里出现两套核心模型，后续维护会变复杂。
7. 如果 Remote 继续绑定 CCLink，直连服务器、内网/VPN 和专业用户场景会被产品形态误伤。
8. 如果远程工作空间列表再套“服务器”父层，会增加认知负担；本地/远程/通道都只能是标识，不是层级。
9. 如果 Codex、Claude Code、DeepInk Agent 被做成不同工作空间类型，后续执行后端切换会变成数据迁移问题。
10. 如果 Terminal 没有工作空间归属和远端权限模型，它会成为最危险也最难恢复的 Tab。
11. 如果保留“远程 Agent 面板”，会把远程会话误导成特殊产品对象；实际上它只是运行在远端的普通会话。
